# 智能体仿真触发事件下拉化设计

## 背景

当前仿真运行区域的“触发事件”使用自由输入框。输入内容只会被写入仿真启动日志和传播队列文本，仿真日志、传播路径和受影响节点不会因为事件不同而发生真实变化。

用户要求将该输入改为下拉选择模式，并且不同触发事件运行后的仿真内容要不同。

## 目标

- 将“触发事件”从文本输入改成下拉框。
- 下拉项基于当前选择的“触发智能体”动态生成。
- 同一智能体选择不同事件后，仿真日志、传播目标和拓扑高亮结果明显不同。
- 无规则智能体不能启动无意义仿真。

## 设计决策

### 1. 事件来源

触发事件直接来源于当前触发智能体已有的规则列表，而不是维护一套独立事件配置。

原因：

- 避免规则配置与仿真事件双份维护。
- 保证仿真入口与规则配置页的数据一致。
- 当前数据模型已经具备 `condition` 和 `actions`，可以直接驱动仿真。

### 2. 事件选择模型

界面状态从 `triggerEvent: string` 改为 `selectedRuleId: string`。

下拉展示文案由规则条件生成，例如：

- `参数变更 | 强度 > 0.7`
- `成本波动 | 材料成本 > 5`
- `约束冲突 | 尺寸 != 兼容`

当用户切换触发智能体时：

- 自动重新计算该智能体的可选规则。
- 默认选中第一条规则。
- 如果该智能体没有规则，则下拉为空并禁用运行按钮。

### 3. 仿真驱动方式

仿真入口不再只是携带事件文本，而是携带被选中的规则对象。

运行时流程：

1. 根据 `triggerAgent + selectedRuleId` 找到规则。
2. 记录规则条件摘要作为启动日志。
3. 顺序解析规则中的 `actions`。
4. 每个动作生成自己的日志、传播目标和传播说明。
5. 将动作解析出的目标智能体加入传播队列，更新拓扑图高亮。

### 4. 动作到仿真行为的映射

为了让不同事件产生不同仿真内容，动作组件将映射为不同的日志和传播行为：

- `影响范围计算`
  - 输出影响等级和分析类日志。
  - 不直接扩展传播目标。
- `通知供应链智能体`
  - 输出通知类日志。
  - 优先将供应链相关智能体加入传播队列。
- `更新设计参数`
  - 输出设计变更日志。
  - 沿通信链路继续传播。
- `BOM版本更新`
  - 输出物料/BOM更新日志。
  - 将受影响后续节点加入传播。
- `冲突消解器`
  - 输出告警和决策日志。
  - 体现冲突处理场景。
- `触发下游智能体规则`
  - 直接读取动作参数中的目标智能体并入队。
  - 形成与普通通信链路不同的传播结果。

### 5. 空状态与异常处理

- 无规则智能体：
  - 事件下拉显示“暂无可选事件”。
  - 禁用“运行仿真”按钮。
- 规则存在但动作无法映射：
  - 写入告警日志。
  - 不中断其余仿真步骤。
- 动作指定的目标智能体不存在：
  - 写入告警日志。
  - 跳过该目标。
- 本次规则没有产生下游传播：
  - 只高亮起始智能体。
  - 输出“本次事件未触发跨智能体传播”。

## 实现拆分

为避免 `Simulation.tsx` 继续堆积复杂逻辑，新增纯函数层：

- 规则转下拉选项文案。
- 根据规则动作解析仿真日志与下游目标。
- 统一执行传播队列并生成拓扑高亮。

UI 组件只负责：

- 当前智能体和规则选择。
- 触发运行。
- 渲染日志与拓扑画布。

## 验证策略

- 触发智能体切换后，下拉列表动态变化。
- 无规则智能体不可运行。
- 同一智能体切换不同规则时，日志内容和影响节点存在差异。
- TypeScript 检查与生产构建通过。

## 风险

- 当前数据量较小，动作映射需要写死一部分业务规则；后续如果动作组件扩展，需要同步补充映射。
- 仿真展示仍然是前端演示型逻辑，不是完整规则引擎；本次目标是让“事件选择”和“仿真结果差异”真实可见。
